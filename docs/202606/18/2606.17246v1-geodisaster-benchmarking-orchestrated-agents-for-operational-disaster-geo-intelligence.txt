Abstract 

Remote-sensing vision-language models (RS-VLMs) have advanced Earth-observation analysis toward visual interpretation and instruction-following, yet fall short of operational geo-intelligence, which demands tool-grounded spatial reasoning and structured, evidence-backed decisions. We introduce GeoDisaster , an operational geospatial disaster reasoning benchmark with 2,921 verified instances across 43 question types and five task families: deforestation monitoring, multi-hazard analysis, building-damage assessment, flood-safe routing, and Sentinel-1 SAR flood monitoring. Instances integrate heterogeneous EO/GIS evidence—optical and SAR imagery, raster masks, vector geometries, road networks, and exposure layers—spanning hazard detection, damage assessment, ex-posure estimation, and diagnostic report generation. Ground-truth answers are grounded in executable geospatial workflows and deterministic consistency checks, removing the need for language-model annotation. We further propose an orchestrated multi-agent framework with 18 disaster-oriented tools, where role-specialized agents coordinate through explicit execution contracts, aligned via 

Role-Contract Expectation Alignment (RCEA) : failure-aware supervised fine-tuning combined with contract-grounded reinforcement learning over dense step-level signals. Experiments show that GeoDisaster challenges existing RS-VLMs and agentic systems, while RCEA improves tool use, evidence grounding, state consistency, and decision generation. https://github.com/VIMAGE-IITB/GeoDisaster 

## 1 Introduction 

Earth observation from satellites and aerial sensors has become a cornerstone of modern disaster man-agement, enabling rapid assessment of floods, wildfires, building damage, and deforestation across large geographic areas. Recent remote-sensing vision-language models (RS-VLMs) have advanced this capability by supporting natural-language interaction, visual interpretation, and instruction-following over Earth-observation (EO) data [ 1 , 2]. Yet these models are fundamentally perception-oriented: they recognize what is visible in an image, answer isolated questions, or generate scene-level descriptions. Operational disaster analysis demands considerably more. A field analyst responding to a flood event must not only detect inundated areas, but also cross-reference satellite and radar observations, estimate exposed infrastructure and population, compute safe evacuation routes, and produce structured situation reports—all under time pressure and across heterogeneous data sources [ 3 , 4 ]. This requires agents to execute multi-step spatial workflows : selecting tools, passing valid inputs, maintaining consistent intermediate state, and synthesizing grounded decisions. Existing RS benchmarks do not evaluate this capacity. Static visual QA datasets assess final outputs from single images, missing procedural correctness, tool-use validity, and multi-step reasoning coherence [ 5, 6]. Recent tool-augmented agentic benchmarks [ 4, 6] advance but remain limited to monolithic single-agent execution, terminal-outcome

## 2 Related Work 

Disaster Geospatial Benchmarks. Early disaster remote-sensing benchmarks focused on static percep-tion: damage labels in xBD/xView2 [ 11 , 12 ], flood masks in Sen1Floods11 [ 13 ], post-flood segmentation in FloodNet [ 5], scene parsing in RescueNet [ 14 ], and flooded-road extraction in SpaceNet 8 [ 15 ]. Subsequent benchmarks expanded to multimodal question answering and geospatial interpretation, in-cluding FloodNet-VQA [ 5 ], VQA-Aid [ 16 ], DisasterM3 [ 17 ], HRVQA [ 18 ], RSVLM-QA [ 19 ], and GeoMMBench [ 20 ]. However, these datasets evaluate final outputs only, missing procedural errors, tool-use validity, and execution coherence [ 21 ]. Recent work has therefore moved toward tool-augmented agentic evaluation: ThinkGeo [ 6] for step-wise tool-augmented RS reasoning, OpenEarthAgent [ 4 ] for supervised geospatial reasoning traces, and GeoMMAgent [ 20 ] for multi-agent geoscience interpretation. RescueADI [ 2 ] introduced adaptive disaster interpretation, while GeoLLM-QA [ 22 ] studied UI-grounded geospatial interaction. GeoDisaster extends this direction to fully operational disaster analysis—covering evacuation routing, damage inventory, SAR/optical deforestation assessment, and multi-temporal flood evolution—where each instance includes structured tool-use trajectories, intermediate state summaries, and raster/vector artifacts, enabling evaluation of both final correctness and the validity of the underlying reasoning process. 

Tool-Augmented and Agentic Geospatial Reasoning. Recent RS agentic systems augment LLMs with domain-specific tools for scene classification and counting [ 23 ], change interpretation [ 24 ], wildfire monitoring [ 25 ], and cross-modal observation analysis [ 26 ]. ThinkGeo [ 6] evaluates ReAct-style agents on structured RS tasks, while OpenEarthAgent [ 4 ] trains geospatial agents with supervised multi-step reasoning trajectories. Trajectory Optimization and Multi-Agent Alignment. LLMs remain unreliable on long-horizon tasks involving tool use and intermediate artifacts [ 4 ]. Post-training approaches address this through supervised trajectory tuning—ToolLLM [ 28 ], AgentTuning [ 29 ]—and preference-based alignment via PPO [ 30 , 31 ], DPO [ 32 ], and GRPO [ 33 ]. Recent extensions target trajectory-level optimization: GiGPO [ 34 ] improves credit assignment through episode- and step-level grouping, SELAUR [ 35 ] introduces uncertainty-aware rewards for multi-step exploration, and GeoReason [ 7] enforces logical consistency between reasoning traces and outputs. In multi-agent settings, failures additionally arise from coordination breakdowns and inconsistent role execution [ 8 ]. MAGRPO [ 36 ], M-GRPO [ 10 ], and MHGPO [ 37 ] extend GRPO-style optimization to collaborative and hierarchical agent systems, while Dr. MAS [ 38 ] stabilizes training through agent-wise normalization. Our Role-Contract Expectation Alignment builds on these foundations by combining failure-aware SFT with contract-conditioned reinforcement learning, explicitly supervising role-conditioned obligations at each step of orchestrated execution—a dimension not addressed by prior trajectory optimization methods.

## 4 Proposed Methodology 

We formulate disaster geospatial reasoning as an orchestrated multi-agent decision process. Given an input task q—comprising a natural-language query, remote-sensing observations, and geospatial context— the objective is to produce a grounded final response y through intermediate planning, tool use, artifact generation, and state-aware synthesis. We define the agent set and trajectory as 

A = {Ai | i ∈ { ORC , GA , VRA , PA }} , τ = {(st, β t, a t, o t, γ t)}Tt=1 . (1) where st is the shared execution state, βt ∈ A is the active agent at step t, at is its action, ot is the resulting observation, and γt is the execution contract governing the current decision. Each role-specific policy is conditioned on the current state and active role instructions, and the final response is generated from the completed trajectory as y = f (τ, q ).We propose a two-stage methodology for operational disaster geospatial reasoning. In the first stage, a contract-driven orchestrated multi-agent framework decomposes complex geospatial tasks across role-specialized agents, where every orchestrator–specialist interaction is governed by a typed execution contract γit that specifies verifiable obligations at each step. In the second stage, Role-Contract Expectation Alignment (RCEA) optimizes agent behavior through failure-aware supervised fine-tuning followed by contract-grounded reinforcement learning over dense, step-level signals. The key insight is that execution contracts—unlike free-form delegation—produce observable, step-level violations vit that enable role-conditioned credit assignment; a property that terminal-reward methods structurally cannot exploit. Together, these stages address the two principal failure modes of existing multi-agent RS systems: (i) unverifiable role execution leading to silent spatial errors, and (ii) misattributed learning signals that destabilize joint optimization. 

4.1 Contract-Driven Orchestrated Agentic Framework 

Our framework adopts a centralized orchestration design in which the orchestrator AORC governs the full execution lifecycle—task decomposition, agent routing, state tracking, recovery, and termination—while three role-specialized agents provide complementary capabilities. Think of AORC as a project manager that assigns subtasks to specialists but never executes them directly. AGA executes geospatial operations (boundary retrieval, distance computation, map rendering); AVRA handles image and raster reasoning (visual interpretation, object localization, segmentation, spectral index analysis); and APA supports numerical computation, filtering, planning logic, and response synthesis. Specialists do not communicate directly—all information exchange is mediated through AORC via a shared, mutable execution state st

that encodes task context, plan status, available artifacts, tool outputs, and execution history. 

Execution contracts as the core design primitive. The central novelty is not multi-agent decomposition per se—which prior systems also employ [ 20 , 27 ]—but the formalization of every orchestrator–specialist interaction as a typed execution contract . In prior RS multi-agent systems, the orchestrator issues free-form instructions: a specialist receives a text description of its task and is free to produce any response. This gives specialists no verifiable obligations—an agent may produce a plausible-sounding but spatially invalid output with no mechanism for detection. 4.2 Failure-Aware Supervised Fine-Tuning

## 5 Experimental Evaluations 

Benchmarks. We evaluate on two agentic RS benchmarks. GeoDisaster assesses disaster-oriented geospatial reasoning under multi-step tool use, constraint satisfaction, and operational decision-making. 

OpenEarthAgent [ 4] evaluates generalization on an existing tool-augmented RS benchmark with broad geospatial task coverage. 

Baselines. We compare closed- and open-source LLMs, the OpenEarthAgent agentic baseline, where applicable, and three variants of our framework. Single-model baselines use structured prompting with tool descriptions but without role decomposition or delegation. Our variants are: MAS —vanilla multi-agent system without alignment; MAS+SFT —with failure-aware supervised fine-tuning applied per agent; and MAS+SFT+GRPO —the full proposed framework with RCEA. 

Fairness Protocol. All methods share the same train/validation/test split, tool registry, tool documentation, execution budget, maximum tool calls, prompts, and stopping criteria. We use Qwen2.5-7B-Inst as the default agent backbone. Both SFT and GRPO are trained on GeoDisaster training instances plus a balanced subset of OpenEarthAgent training data. 

Evaluation Metrics. We report both end-to-end and step-wise metrics to evaluate full trajectory quality beyond final-answer correctness. End-to-end metrics cover tool-sequence fidelity under order-agnostic, exact-order, and unique-tool matching ( ToolAnyOr , ToolSameO , ToolUni ), task success ( TSR ), answer correctness ( Ans ), generative output quality ( Gen ), contract satisfaction ( CSR ), and strict episode comple-tion ( ESR ). Step-wise metrics cover instruction following ( Inst. ), tool selection ( Tool ), argument-name completeness ( ArgN ), argument-value correctness ( ArgV ), and execution summarization ( Summa ). All metrics are scored using GPT-5.5 as an LLM-as-judge; Deeper description provided in the supplementary.

## 6 Conclusions 

We presented GeoDisaster , an operational geospatial disaster reasoning benchmark with 2,921 verified instances grounded in executable geospatial workflows, and RCEA , a two-stage alignment protocol for orchestrated multi-agent execution. RCEA combines failure-aware SFT with contract-grounded reinforcement learning to address credit misattribution and reward-scale heterogeneity through dense step-level signals. Experiments show that existing RS-VLMs fall substantially short of operational geo-intelligence, while our aligned framework achieves strong performance across final-answer and trajectory-level metrics on both GeoDisaster and an established remote-sensing agentic benchmark. Failure-wise analysis further shows that SFT improves contract-compliant execution, while GRPO enhances trajectory stability and decision consistency. Overall, the gains across both benchmarks indicate that RCEA supports reliable tool-grounded multi-agent reasoning beyond the proposed dataset. Future work should further improve uncertainty-aware reasoning over multi-source and multi-evidence geospatial settings.

S1.2 Geospatial Tool Suite 

To support tool-grounded disaster reasoning, GeoDisaster defines a compact suite of 18 geospatial tools, summarized in Table S2. These tools abstract dataset-specific operations into reusable reasoning functions covering evidence loading, spatial measurement, temporal and cross-source comparison, impact assessment, route planning, and artifact synthesis. This design serves two purposes. First, it encourages evidence-grounded problem solving, where agents must issue valid tool calls, pass correct handles and arguments, and use intermediate observations rather than relying only on direct answer generation. Second, it enables fine-grained evaluation of where an agent fails: evidence access, spatial computation, tool choice, argument construction, cross-source reasoning, decision planning, or final synthesis. Thus, the tool suite provides both the operational substrate for GeoDisaster tasks and the basis for trajectory-level evaluation beyond final-answer correctness. 

S1.3 Representative Error Types 

Beyond task-family diversity, GeoDisaster provides trajectory-level supervision for both single-agent and multi-agent execution. Prior agentic systems commonly suffer from structural and semantic failures that are not fully captured by final-answer accuracy, such as invalid planning, wrong tool selection, incorrect argument passing, premature termination, and inconsistent synthesis. By analyzing trajectories across different stages of training, fine-tuning, and alignment, we identify the recurring failure modes summarized in Table S3. These failures occur at different levels of the execution process. Planning and delegation errors affect the global workflow before tool execution begins, while tool, argument, and format errors arise during local action generation. Termination, loop, and replanning errors reflect failures in state tracking and control-flow management. This taxonomy motivates our alignment strategy: failure-aware SFT provides corrected recovery actions under the same state and contract, while GRPO encourages contract-consistent execution and penalizes invalid or inefficient trajectories. While our framework significantly reduces these errors quantitatively, several challenges remain, especially for ambiguous visual evidence, invalid geospatial arguments, and premature stopping. These examples suggest that future improvements should focus on uncertainty-aware replanning, stronger argument validation, and better use of multi-temporal evidence.

S2.2 Role-Contract GRPO Alignment 

After SFT, GRPO is used to refine trajectory-level decisions that are difficult to capture through imitation alone. The reward combines final task success with role-specific contract satisfaction, robustness, and efficiency terms. Contract checks include schema validity, handle existence, spatial overlap or containment, route–flood intersection, area/count tolerances, and required-field coverage, with borderline spatial cases resolved using dataset-resolution and task-specific tolerance thresholds. Rewards are normalized within each role, so the orchestrator and specialist agents are updated according to their own execution distributions rather than a shared global scale. This is important because ORC, GA, PA, and VRA differ in activation frequency, action type, and failure modes. Figure S3 shows that role-wise rewards increase and then stabilize during GRPO alignment, indicating that the policy improves without collapsing the distinct behavior of different agents. GA and PA obtain higher rewards because many tasks involve measurable geospatial operations and analytic decisions, while ORC and VRA show more moderate but stable improvements in coordination and evidence interpretation. S3.3 LLM-as-Judge Evaluation Protocol 

We use GPT-5.5 as an LLM-assisted evaluator for GeoDisaster trajectories. The evaluator is given the complete saved case, including the user question, visible input context, expected answer, reference tool chain or plan, predicted final answer, full predicted trajectory, tool calls, tool observations, runner metrics, and failure metadata. Importantly, the judge is instructed to inspect the full execution trace and separate deterministic trace-based metrics from semantic and hybrid judgments. To check reliability, we manually reviewed a subset of 50 evaluated cases and revised the prompt until the produced metric judgments were consistent with human inspection. Figure S9 shows condensed LLM-as-judge prompt used for our evaluations. The evaluation uses two main metric groups. End-to-end metrics measure whether the full trajectory reaches the intended task outcome. ToolAnyOr , ToolSameO , and ToolUni evaluate tool-chain fidelity under order-agnostic multiset matching, exact ordered matching, and unique-tool coverage, respectively. TSR 

summarizes overall task success by combining final-answer quality, episode completion, tool coverage, and tool-order fidelity. Ans measures task-level correctness for non-generative outputs, while Gen 

measures artifact-generation quality when the task requires a map, image, or visual output. CSR measures whether the final answer satisfies schema, grounding, units, source, temporal, and task-specific constraints. 23 ESR captures clean episode completion, penalizing premature stopping, unresolved failures, harmful loops, weak synthesis, and invalid plans. For multi-agent settings, PlanAcc measures the quality of the orchestrator’s decomposition, and DelegAcc evaluates whether subtasks are assigned to the appropriate specialist agents. 

Step-wise execution metrics evaluate the internal quality of the trajectory. Inst measures whether actions are syntactically valid, logically executable, and consistent with the required control format. 

Tool measures local tool or control-action correctness at each step, which differs from whole-trajectory tool-chain fidelity. ArgN measures the presence of required tool parameters, while ArgV measures the correctness of parameter values, including handles, regions, modalities, timestamps, source selectors, thresholds, and task-specific options. Summ measures the quality of grounded intermediate completion or handoff, including whether the produced summary is concrete, evidence-supported, schema-aware, and useful for downstream reasoning. It is therefore distinct from Ans , which measures final-answer correctness against the expected answer. The prompt further distinguishes three scoring modes. Deterministic/trace metrics are computed directly from the reference chain, predicted tool calls, runner events, and visible observations; these in-clude tool-chain fidelity, strict correctness, most instruction/tool/argument checks, raw episode resolution, and explicit failure tags. Semantic metrics , such as Ans , Summ , CSR , PlanAcc , and DelegAcc , require interpretation of the final answer, evidence use, schema compliance, and trajectory meaning. Hybrid metrics , such as TSR , final ESR , final Tool , final ArgV , and role-level scores, combine trace verification with semantic judgment when valid equivalent actions, partial execution, or backend/tool failures require contextual interpretation. 27 GeoDisaster LLM-as-Judge Prompt (condensed) 

GeoDisaster LLM-as-Judge Prompt 

You are an evaluation assistant responsible for fair, strict, tool-aware evaluation of GeoDisaster geospatial agent conversations. You will receive one saved case containing the user question, visible input context, expected final answer, GT/reference tool chain or plan when available, predicted final answer, full predicted trajectory, tool calls, tool observations, runner metrics, and failure metadata. Read the full case before scoring. Evaluate the agent’s behavior from the visible trajectory and evidence, not hidden data. Runner metrics may be used as evidence, but verify them against the trace and GT; if they conflict, score from the trace and briefly explain the discrepancy. Use the same metric definitions for single-agent and multi-agent runs: • Multi-agent runs should include per-role scores for visible roles: `ORC`, `GA`, `PA`, and `VRA`. • Single-agent runs should include one `MAIN` role score. 

Metric families: 

• Deterministic/trace: `AnyOr`, `SameO`, `Uni`, `StrictCorrect`, most of `Inst`, `Tool`, `ArgN`, `ArgV`, raw `ESR`, and explicit failure tags. Compute these directly from the GT chain, predicted trajectory, tool calls, runner events, and visible observations. • Semantic judge: `Ans`, `Summ`, `CSR`, semantic `PlanAcc`, semantic `DelegAcc`, and synthesis/failure severity. Score these from the final answer, evidence, schema compliance, and trajectory meaning. • Hybrid: `TSR`, final `ESR`, final `Tool`, final `ArgV`, and role-level scores when valid equivalent actions, partial execution, or backend failures require both trace verification and semantic judgment. 

Trajectory normalization: 

• Build `expected_tools` from the GT/reference tool chain and `actual_tools` from executable predicted tool/API calls. • Exclude pure planning, termination, and final-answer text unless explicitly part of the target action. • Deduplicate only immediate duplicate retry/log entries with identical tool and arguments. • Preserve multiplicity for `AnyOr`, preserve order for `SameO`, and ignore multiplicity for `Uni`. 

Metric definitions: 

• `AnyOr`: order-agnostic multiset tool-chain fidelity. Score 1.0 only if all expected tools appear with required multiplicity, ignoring order. Extra calls are allowed. • `SameO`: ordered tool-chain fidelity. Score 1.0 only if `AnyOr=1.0` and the expected tool sequence appears as an ordered subsequence of `actual_tools`. • `Uni`: unique required-tool coverage. Score 1.0 only if every distinct expected tool appears at least once. • `Inst`: syntactic and logical action validity, including valid action format, legal ORC events, correct delegation/termination, and absence of invalid loops or impossible state references. • `Tool`: local step-wise tool/control correctness for the current state and subtask. This differs from `AnyOr`, which scores whole-trajectory tool coverage. • `ArgN`: required argument-name coverage for emitted tool calls, including mandatory fields and expected nesting/shape. • `ArgV`: correctness of argument values, including handles, regions, modalities, timestamps, labels, thresholds, source selectors, and task-specific options. • `Ans`: semantic final-answer correctness independent of formatting. Accept numeric values within +/-10% unless exact IDs, counts, ordering, schema, or strict values are required. • `StrictCorrect`: strict runner-style correctness. Score 1.0 only when the final answer exactly satisfies the required exact, numeric, or numeric-subset checks. • `CSR`: constraint satisfaction, including answer schema, required fields, units, region/time/source constraints, grounding, and output format. • `ESR`: clean episode completion without agent-caused abort, premature stop, unrecovered tool/backend failure, unresolved missing evidence, or harmful loops. • `PlanAcc`: planning/decomposition quality. For multi-agent runs, judge ORC’s plan against the GT plan and tool dependencies; for single-agent runs, judge the implicit or explicit plan. • `DelegAcc`: specialist-assignment correctness. ORC coordinates; GA handles geospatial/vector/raster context; VRA handles visual, damage, flood, and scene interpretation; PA handles statistics, ranking, exposure, route decisions, classification, and final analytic synthesis. Return `null` for single-agent runs unless delegation-like behavior is present. • `Summ`: grounded completion or handoff quality. Score whether the response is supported, complete, schema-aware, readable, and useful. This is separate from `Ans`. • `TSR`: aggregate task success. Use the mean of `Ans`, `CSR`, `ESR`, `AnyOr`, and `SameO` as the default, then adjust by at most +/-0.10 for severe hallucination, missing evidence, or unusually strong evidence-backed completion. • `Gen`: artifact-generation quality. Return `null` when no artifact is requested; otherwise score whether the requested artifact is produced and usable. 

Backend/tool policy: 

• Do not mark `ArgErr` for backend timeout, weak segmentation/detection, or unstable server output if the selected tool and arguments were correct. • Tag backend failures as `ToolExecErr` or `ToolErr` only if they materially affected the trajectory. • Do not inflate `Ans` or `Summ` if the agent fabricated a final answer without sufficient evidence. • If tools produced enough evidence for synthesis but the final response failed to use it, penalize `Summ` and, when appropriate, `Ans`. 

Failure tags: 

• `PlanErr`: missing, invalid, or insufficient planning. • `AgentErr`: wrong specialist choice, skipped role, role inconsistency, or role-contract violation. • `ToolErr`: missing required tool, wrong tool family, unsuitable tool choice, or incomplete tool-chain coverage. • `ArgErr`: missing, wrong, inconsistent, or non-value-preserving arguments. • `FormatErr`: malformed JSON/action/output schema or invalid action wrapper. • `TermErr`: premature, missing, or invalid termination. • `LoopErr`: redundant calls or repeated reasoning loops that affect completion. • `ReplanErr`: bad, unnecessary, contradictory, or missing recovery/replanning. • `SynthErr`: weak final synthesis despite available evidence. • `ConstraintErr`: schema, modality, source, region/time, unit, grounding, or task-constraint violation. • `ToolExecErr`: live tool failure, timeout, failed observation, or backend/control failure affecting the trajectory. • `AbortErr`: no clean completion or no usable final answer. 

Output requirements: 

Return strictly one JSON object and nothing else. 

The JSON object must include: 

• `case_id` and `mode`, where `mode` is `multiagent` or `singleagent`. • All applicable case-level metrics using the keys defined above. Report numeric scores as floats in `[0.000, 1.000]`; use `null` only when a metric is not applicable. • Boolean status fields: `answer_correct`, `strict_correct`, `constraint_satisfied`, and `execution_success`. • `deterministic_audit` containing `expected_tools`, `actual_tools`, and a short discrepancy note when needed. • `agent_scores`, where each visible role (`ORC`, `GA`, `PA`, `VRA`) or `MAIN` contains the same applicable metric keys as floats in `[0.000, 1.000]`, plus `turn_count` and `tool_call_count`. • `failure_tags`, `dominant_failure`, and a concise `justification` of 2--4 sentences. 

Scoring reminders: 

• Compute deterministic trace metrics first, then semantic metrics from the final answer and evidence. • Use `answer_correct=true` when `Ans >= 0.800`. • Use `strict_correct=true` only when `StrictCorrect >= 0.950`. • Use `constraint_satisfied=true` when `CSR >= 0.800`. • Use `execution_success=true` when `ESR >= 0.800`. • Always include `agent_scores`; use `MAIN` for single-agent runs. Figure S9: Condensed LLM-as-judge prompt used for GeoDisaster evaluation. The prompt defines deterministic, semantic, and hybrid metric families; specifies scoring rules for tool-chain fidelity, final-answer correctness, constraint satisfaction, execution success, role-level agent scores, and failure tags; and enforces structured JSON output for reproducible aggregation.
