# No Dense Tensors Needed: Fully Sparse Object Detection on Event-Camera Voxel Grids 

## Mohamad Yazan Sadoun 1, Sarah Sharif 1, Yaser Mike Banad 1* 

1* University of Oklahoma, Norman, OK, USA. *Corresponding author(s). E-mail(s): bana@ou.edu; Contributing authors: mohamad.yazan@ou.edu; s.sh@ou.edu; 

Abstract 

Event cameras produce asynchronous, high-dynamic-range streams well suited for detecting small, fast-moving drones, yet most event-based detectors convert the sparse event stream into dense tensors, discarding the representational effi-ciency of neuromorphic sensing. We propose SparseVoxelDet, to our knowledge the first fully sparse object detector for event cameras, in which backbone feature extraction, feature pyramid fusion, and the detection head all operate exclusively on occupied voxel positions through 3D sparse convolutions; no dense feature ten-sor is instantiated at any stage of the pipeline. On the FRED benchmark (629,832 annotated frames), SparseVoxelDet achieves 83.38% mAP@50 while processing only ∼14,900 active voxels per frame (0.23% of the T ×H×W grid), compared to 409,600 pixels for the dense YOLOv11 baseline (87.68% mAP@50). Relaxing the IoU threshold from 0.50 to 0.40 recovers mAP to 89.26%, indicating that the remaining accuracy gap is dominated by box regression precision rather than detection capability. The sparse representation yields 858 × GPU memory com-pression and 3,670 × storage reduction relative to the equivalent dense 3D voxel tensor, with data-structure size that scales with scene dynamics rather than sen-sor resolution. Error forensics across 119,459 test frames confirms that 71% of failures are localization near-misses rather than missed targets. These results demonstrate that native sparse processing is a viable paradigm for event-camera object detection, exploiting the structural sparsity of neuromorphic sensor data without requiring neuromorphic computing hardware, and providing a frame-work whose representation cost is governed by scene activity rather than pixel count, a property that becomes increasingly valuable as event cameras scale to higher resolutions. 

Keywords: Event camera; Sparse 3D convolution; Drone detection; Object detection; Event-based vision; Sparse voxel representation

# 1 Introduction 

Detecting small unmanned aerial vehicles (drones) in real time is critical for airspace security, yet conventional RGB-based detectors struggle under low-light, high-dynamic-range, and cluttered-sky conditions [1]. Event cameras, neuromorphic sensors that asynchronously report per-pixel brightness changes with microsecond latency and 

>120 dB dynamic range [2], are increasingly used for this task [3]. Recent benchmarks such as FRED [1] demonstrate that dense detectors applied to event-frame represen-tations already achieve strong accuracy (87.68% mAP@50 with YOLOv11), validating the information content of the event stream for drone detection. However, converting asynchronous events into dense frame-like tensors fundamen-tally discards the sensor’s most distinctive property: extreme sparsity . On the FRED benchmark, typical event frames occupy less than 1% of the input voxel grid, yet dense detectors must process all 409,600 pixel positions after resizing to 640 ×640. This dense conversion imposes a computational cost disproportionate to the actual information content, undermining the efficiency promise of neuromorphic sensing (Figure 1). This mismatch creates a central opportunity for neuromorphic perception: if com-putation is tied to event activity rather than image resolution, event-camera systems can scale to higher resolutions and longer deployments without inheriting the full cost profile of dense vision pipelines. In the LiDAR domain, sparse 3D convolution backbones such as SECOND [4] and detection frameworks including PointPillars [5] and CenterPoint [6] have progressively reduced the reliance on dense representations. Most recently, VoxelNeXt [7] demon-strated that point-cloud object detection can be performed entirely through sparse 3D convolutions without constructing any dense bird’s-eye-view maps. We observe that event-camera voxel grids share a similar sparse structure: both are 3D tensors (tem-poral bins × height × width for events; depth × height × width for LiDAR) with low occupancy rates. This structural analogy motivates our central hypothesis: can an object detector process event data natively in the sparse domain, end-to-end, without any dense tensor conversion? 

We present SparseVoxelDet, to our knowledge the first fully sparse object detector for event cameras. Existing event detectors either convert events to dense frames before applying conventional architectures [8–10] or employ sparse representations only par-tially, reverting to dense feature maps at intermediate stages [11, 12]. SparseVoxelDet maintains sparse 3D tensor representations end-to-end, reframing event-camera detec-tion as a native sparse-computation problem rather than a dense-image adaptation. The architecture chains three purely sparse stages: (1) a SparseSEResNet backbone that processes event voxel grids through 3D sparse convolutions with element-wise residual connections and squeeze-and-excitation attention, producing multi-scale fea-tures at strides 4, 8, and 16; (2) a Sparse Feature Pyramid Network that fuses these scales top-down through sparse lateral connections and sparse transpose convolu-tions, producing a unified feature map at stride 4; and (3) a SparseDetHead that applies an MLP to each active voxel position to predict classification, box regression (LTRB), and centerness, following the anchor-free FCOS formulation [13]. After tem-poral max-pooling to collapse the time dimension, non-maximum suppression yields final detections. No dense feature map is allocated at any point. 2On the FRED canonical drone detection benchmark [1], SparseVoxelDet achieves 83.38% mAP@50 with 6.2M parameters, narrowing the gap to the YOLOv11 baseline (87.68%) to 4.3 percentage points. To understand the nature of this gap, we conduct an error forensics analysis across all 119,459 test frames. The analysis reveals: (i) 71% of false negatives are localization near-misses (IoU > 0 but < 0.5), indicating the model detects the drone but regresses imprecise boxes; (ii) confidence separation is strong (μTP = 0 .46 vs. μFP = 0 .11); and (iii) recall reaches 95.4% at IoU ≥ 0.40, confirming that the gap is predominantly a box-tightness penalty rather than a detection failure. Our contributions are twofold: 1. Fully sparse architecture. We introduce SparseVoxelDet, an event-camera object detector that performs all computation in the sparse voxel domain. The model processes ∼14,900 active voxels per frame (28 × fewer than dense baselines) and achieves 83.38% mAP@50 (within 4.3 points of the dense baseline), with 858 ×

GPU memory compression and 3,670 × storage reduction relative to the equivalent dense 3D voxel tensor. While dense detector cost grows as O(H × W ) with sensor resolution, SparseVoxelDet’s cost is proportional to scene activity: the active voxel count increases by only 9% when moving from 640 2 to native 1280 × 720, despite a 2.25 × increase in pixel count. The architecture uses a 6-channel temporal surface feature representation incorporating recency and temporal variation that enables operation at any sensor resolution without architectural modification. 2. Error forensics and empirical analysis. Through a systematic analysis pipeline applied across the FRED drone detection benchmark, we decompose detection fail-ures into actionable categories (localization error, confidence ranking, object size, event density) and identify box regression precision, an inherent challenge of sparse representations that lack dense neighborhood context, as the primary performance bottleneck. This provides a sparse-native evaluation on FRED and a reusable forensics methodology applicable to any single-class detector. 

# 2 Related Work 

## 2.1 Event-Based Object Detection 

Event cameras [2] report pixel-level brightness changes asynchronously, producing streams with microsecond temporal resolution and high dynamic range [3]. Adapting object detectors to this modality has followed two main strategies. Dense conversion methods. The predominant approach converts events into frame-like representations (event histograms [14], time surfaces, or voxel grids [15]) and applies conventional dense detectors. Perot et al. [8] introduced the first large-scale event detector on the 1-megapixel Prophesee Gen1 dataset using histogram-based representations. Messikommer et al. [16] proposed learning event representations end-to-end for object detection rather than hand-crafting them. The FRED benchmark [1] reports results for YOLOv11 [17], RT-DETR [10], and Faster R-CNN operating on event frames extracted at 30 FPS. These methods achieve strong accuracy but inherit the computational cost of dense convolutions, processing all spatial positions regardless of event occupancy. Sparse event processing. ER-DETR [22] fuses RGB and event modalities via cross-attention for drone detection. However, these architec-tures typically convert to dense intermediate representations during feature fusion or detection head stages. Our work extends the spiking residual idea by maintaining full sparsity throughout: we use the SEW residual pattern with sparse 3D convolutions rather than dense spiking neurons. 42.2 Sparse 3D Convolutions for Point-Based Detection 

Sparse convolutions, pioneered by Graham et al. [23] and implemented efficiently in libraries such as spconv [24], process only occupied voxel positions in 3D grids. This approach is well-established for LiDAR-based 3D object detection, where point clouds are voxelized before processing. Early point-based approaches such as PointNet [25] and PointNet++ [26] processed raw point sets directly, but voxel-based methods have since become dominant for efficiency. SECOND [4] introduced sparse convolutions for LiDAR detection, PointPillars [5] organized point clouds into vertical columns for efficient processing, and CenterPoint [6] combined sparse backbones with center-based detection heads. In the 2D domain, Sparse R-CNN [27] demonstrated that a small set of learnable proposals can replace dense feature maps for detection, though it relies on a dense backbone. VoxelNeXt [7] demonstrated that 3D object detection on LiDAR point clouds can be performed fully sparsely , predicting bounding boxes directly from sparse voxel features without any dense bird’s-eye-view conversion. This eliminates the computa-tional overhead of processing empty space. The key insight that sparse-in, sparse-out architectures can match dense counterparts when occupancy is low directly motivates our work on event cameras, where we measure occupancy rates below 1% on FRED (Section 3.1), making dense processing particularly wasteful. Our SparseFPN adapts the top-down feature pyramid concept from FPN [28] to the sparse domain, using sparse transpose convolutions for upsampling and sparse addition for multi-scale fusion. Unlike dense FPN implementations, our Sparse FPN propagates features only to positions that have evidence from at least one scale. 

## 2.3 Drone Detection 

Detecting small unmanned aerial vehicles is a growing security concern that poses unique challenges: drones are small (often <20 pixels), move fast, and operate under variable lighting including dawn, dusk, and night conditions. RGB-based approaches [29] struggle under these conditions due to low contrast and motion blur. The FRED dataset [1] provides synchronized RGB and event data for drone detec-tion across 231 sequences with diverse scenarios including varying altitude, weather, and illumination. It establishes three benchmarks: detection, tracking, and trajectory forecasting. Event-frame representations significantly outperform RGB (+52 mAP@50 points for YOLOv11 on canonical splits), confirming that event cameras capture motion-salient features invisible to RGB under challenging conditions. Our work contributes sparse-native results on FRED, bridging the gap between the event-based detection literature (which has not used sparse 3D convolutions) and the sparse convolution literature (which has focused exclusively on LiDAR).

## 3.2 SparseSEResNet Backbone 

The backbone is a sparse 3D convolutional residual network whose element-wise residual connections are inspired by SEW-ResNet [21], adapted to operate entirely through sparse convolutions following the deep residual learning paradigm [30]. Unlike SEW-ResNet, our network does not use spiking neurons; the residual pattern is adopted purely for its gradient-friendly properties in deep sparse networks. It uses the 

nano deep configuration: a base channel width of 32 with block depths [2 , 2, 2, 1] across four stages, and squeeze-and-excitation (SE) attention [31] on the final two stages. Architecture. A sparse convolutional stem with stride (1 , 2, 2) maps the Cin -channel input to 32 channels while downsampling spatially by 2 ×. Four stages of sparse residual blocks follow with per-stage strides (1 , 2, 2, 2), producing cumulative spatial strides 2, 4, 8, and 16 from the original input (equivalently 1, 2, 4, 8 relative to the stem output). Feature channels double at each stage: 32, 64, 128, 256. Each residual block consists of two SubMConv3d (submanifold sparse convolution) layers with layer normalization and ReLU activation, connected by an element-wise residual addition that operates on the sparse feature tensors. Residual connections. Following the element-wise addition pattern of Fang et al. [21], residual connections add features in the sparse domain. When the input and output dimensions differ (at stride boundaries), a 1 × 1 sparse convolution with stride acts as the shortcut. The backbone outputs multi-scale features at three levels: {C2, C 3, C 4} with stride 

{4, 8, 16 } and channels {64 , 128 , 256 }. All operations are sparse: the number of active positions decreases at deeper stages as sparse strided convolutions subsample only occupied locations.

## 3.5 Detection Head 

The detection head follows the anchor-free FCOS design [13], predicting classification, box regression, and centerness at each active 2D position. Since the temporally-pooled features are a flat tensor of M ′ position vectors (not a dense grid), the head is imple-mented as a shared MLP (equivalent to pointwise 1 × 1 convolutions), operating independently per position. This pointwise design is a deliberate choice: spatial con-text has already been aggregated by the backbone’s multi-layer 3D sparse convolutions 8and the FPN’s top-down fusion; the head’s role is to classify and localize from the resulting per-position feature vector. Shared trunk. Two layers of Linear → GroupNorm → ReLU, both with hidden dimension d = 128. Prediction branches. Three linear heads applied in parallel: 

• Classification: ˆ c ∈ RM ′×1, sigmoid-activated, initialized with prior probability p0 =0.01.

• Centerness: ˆ z ∈ RM ′×1, sigmoid-activated quality estimate following Tian et al. [13]. Box decoding. Each active position ( y, x ) at stride s = 4 corresponds to a region center at pixel coordinates ( x · s + s/ 2, y · s + s/ 2). ## 4.2 Event Preprocessing 

The preprocessing pipeline converts raw asynchronous event streams into sparse voxel grids suitable for 3D sparse convolution. We evaluate two spatial resolutions sharing identical architecture and 6-channel temporal surface features: 

• 640 × 640 (primary): Cin =6 temporal surface features (Section 3.1), T =16 temporal bins, spatially resized to 640 ×640, hot pixel filtering (pixels whose total event count exceeds 3 × the per-pixel mean across the recording are suppressed as sensor noise). 11 This is the primary configuration, selected because it achieves the highest accuracy due to denser voxel grid occupancy (Section 5.1). 

• Native 1280 ×720 (ablation): Same 6-channel features and T =16 bins, at the sensor’s native resolution (no spatial rescaling). This isolates the effect of spatial resolution while controlling for feature representation. The preprocessing pipeline handles the complete transformation from raw asyn-chronous events to clean sparse voxel grids: hot pixel identification and removal, temporal quantization into uniform bins, extraction of six statistical features per active voxel (event counts, timestamps, and temporal variation per polarity), sparse coordinate assembly, and YOLO-format label alignment with sub-frame temporal correspondence. This pipeline is tailored to the FRED dataset’s EVK4 sensor charac-teristics and annotation protocol; adapting it to other event camera datasets would require re-engineering the temporal alignment, resolution handling, and label format conversion stages. Common pipeline. The continuous event stream is divided into non-overlapping windows of 33 ms (one frame at 30 FPS), aligned with FRED annotation timestamps. ## 4.4 Training Details 

SparseVoxelDet is trained with a cosine learning rate schedule over a maximum of 50 epochs, with early stopping based on validation loss. We report results from the best EMA checkpoint (epoch 4, selected by minimum validation loss) for our primary 640 ×640 configuration, and a native 1280 ×720 resolution ablation to assess resolution sensitivity. Figure 5 shows the training trajectory through epoch 7, at which point validation loss had risen for 3 consecutive epochs while mAP@50 declined from 83.38% to 82.90%, confirming the early stopping decision. Training loss continues to decrease, indicating that the model can still reduce training error but at the cost of overfitting; the divergence between training and validation loss after epoch 4 is the standard signal for early stopping. 

• Optimizer: AdamW [34] with learning rate 3 × 10 −4 and weight decay 1 × 10 −2.

• Schedule: Cosine annealing with 5,000-step linear warmup. 

• Batch size: 2 per GPU (single-GPU training). 

• Precision: Mixed-precision training (FP16) with dynamic loss scaling. 

• EMA: Exponential moving average of weights with decay 0.9997. 

• Gradient clipping: Max norm 5.0 with NaN gradient sanitization. 

• Loss weights: λcls = 1 .0, λreg = 2 .0, λctr = 1 .0. 

• Focal loss: α = 0 .25, γ = 2 .0. 13 • Augmentation: Horizontal flip ( p=0 .5), polarity inversion ( p=0 .3, permuting the 6-channel feature pairs), random event dropout ( p=0 .02), random scale augmentation (×0.8–1.2). Multi-scale training cycles the input size among {576 2, 608 2, 640 2} every 10 batches for the 640 2 configuration (and proportional rectangular sizes for native resolution). 

• Multi-seed: Both configurations (640 2 and native 1280 × 720) are trained with seeds 42 and 123 to assess variance. Training is performed on a single NVIDIA RTX 3090 (24 GB) GPU. Each epoch processes all ∼510,000 training frames (batch size 2), yielding approximately 255,000 gradient updates per epoch. The early convergence at epoch 4 reflects the scale of this dataset: 4 epochs correspond to approximately 1,020,000 parameter updates, com-parable to training schedules used in similarly sized dense detectors on datasets of equivalent annotation density. The divergence between training and validation loss after epoch 4 confirms the early stopping decision. Validation is performed every epoch using the EMA weights. Both configurations are trained with seeds 42 and 123; mAP@50 variance is ±0.16 pp at 640 2 and ±0.14 pp at native resolution, confirming low stochastic variance. The primary results (Table 2) report the best seed (42). We adopt standard hyperparameters for the detection framework: focal loss param-eters ( α=0 .25, γ=2 .0) and loss weights ( λcls =1, λreg =2, λctr =1) follow FCOS [13]. 

## 4.5 Evaluation Protocol 

We report the standard COCO-style mAP@50 and mAP@50:95 metrics computed over all test frames. At inference, the score threshold is 0.05 (a deliberately permis-sive value standard in COCO-style evaluation that retains all plausible detections for the precision–recall curve computation; the AP metric integrates over all confidence thresholds, so this floor has negligible impact on reported mAP), NMS IoU threshold is 0.5, and the maximum number of detections per frame is 100. The evaluation uses the EMA model weights.

## 5.1 Main Results 

Table 2 presents detection results on the FRED canonical split (119,459 test frames). SparseVoxelDet processes only ∼14,900 active voxels per frame, 28 × fewer spatial positions than the dense methods that process every pixel. These results are achieved with mixed-precision (FP16) training on a single GPU. No model in the FRED baseline set operates on native sparse representations; all convert event frames to dense images before processing. The 4.3-point mAP@50 gap is predominantly a box-tightness penalty : the model sees the drone in nearly every frame, but its predicted box is 5–10% too loose. This pattern is structurally linked to how event cameras encode moving objects. Unlike conventional cameras that capture the complete spatial extent of targets as filled regions, event cameras fire only at pixels experiencing brightness changes, 19 generating contour-like activations concentrated along motion boundaries. A drone translating against a static background produces events primarily along its leading and trailing edges, with few events in its interior where brightness remains constant. The resulting sparse voxels trace the object’s silhouette rather than filling its area. Since SparseVoxelDet regresses bounding boxes from these occupied positions alone, with no dense spatial context between active voxels, the network must infer object extent from incomplete boundary evidence. Dense detectors, by contrast, process every pixel, including the uniform object interior, providing a clear spatial template for box regression. This asymmetry explains why detection recall is high (the contour signal is sufficient to recognize the drone) while box precision suffers (the contour signal underspecifies the object’s full spatial extent). The effect intensifies for faster-moving targets: rapid motion produces elongated event trails along the trajectory, potentially biasing the regressed box to include the motion streak rather than the instantaneous object footprint. 

## 5.3 Representation Efficiency 

One important advantage of fully sparse processing is in representation : Spar-seVoxelDet operates directly on the sparse event stream without reconstructing dense frames, yielding order-of-magnitude reductions in data movement, storage, and memory footprint. At 640 × 640, the median frame contains only ∼14,900 active voxels out of 16 ×640 ×640 ≈ 6.6 million possible positions (0.23% occupancy). At native 1280 ×720, occupancy drops to 0.11% ( ∼16,200 voxels out of 14.7 million possible positions). Storing only the occupied coordinate-feature pairs requires ∼94 KB of GPU memory per frame, compared to ∼78.6 MB for the equivalent dense 3D tensor ( T ×H×W with float32 features), an 858 × compression. On disk, the sparse NPZ representation aver-ages ∼22 KB per sample versus ∼78.6 MB dense, a 3,670 × storage reduction. These ratios quantify the data-structure efficiency of storing a 3D voxel grid sparsely ver-sus densely. For practical context: the FRED baselines process 2D event-frame images (a single H×W plane, typically ∼1.2 MB at 640 2 in float32), not the full 3D grid. Against this 2D baseline input the sparse advantage is still substantial ( ∼13 × in mem-ory, ∼55 × on disk) and, crucially, SparseVoxelDet retains the full T =16 temporal axis that 2D conversion discards. The sparse representation’s memory footprint scales with scene activity (the number of brightness-change events) rather than sensor resolution, a property with direct implications for edge deployment on memory-constrained plat-forms. At dataset scale (629,832 frames), the full FRED training set occupies ∼8.7 GB in sparse format; the dense equivalent would require ∼40 TB. Dense detector cost scales as O(H × W ): YOLOv11 must process 409,600 pixels at 640 2 and 921,600 at 1280 × 720 (2 .25 × more). In contrast, the sparse representation’s size is governed by scene dynamics, not sensor resolution. We measure a median of 

∼14,900 active voxels at 640 2 and ∼16,200 at native 1280 × 720, a mere 9% increase despite a 2 .25 × growth in pixel count. For future 4K event sensors (3840 ×2160), dense cost would grow ∼20 ×; sparse cost remains bounded by the number of brightness-change events in the scene. 20 Dense detectors applied to event cameras convert the asynchronous event stream into image-like tensors. The FRED baselines use simple event accumulation (histogram binning), which is computationally light. However, applications requiring high-quality frame reconstruction (e.g., E2VID [36], 15–30 GFLOPs; FireNet [37], 1.5–3 GFLOPs) face substantially higher overhead. SparseVoxelDet bypasses any such conversion: vox-elization is a scatter operation with O(Nevents ) additions and zero multiply-accumulate cost. No dense feature map is allocated at any stage. At deeper layers, strided sparse convolutions subsample only occupied positions, reaching ∼2,700 active 3D posi-tions at stride 16 versus ∼24,000 in the dense equivalent. For edge deployment on memory-constrained platforms (e.g., Jetson Orin Nano with 8 GB shared RAM), this compression is critical: the sparse pipeline can process high-resolution event streams that would be infeasible for dense methods under the same memory budget. The sparse backbone processes only occupied voxels and their immediate neigh-bors, achieving compute proportional to the number of active positions. However, the current Feature Pyramid Network uses sparse transposed convolutions for upsam-pling, which expand the set of active positions at each fusion stage. All computations remain on sparse tensors throughout—no stage reverts to dense processing—but this expansion partially attenuates the sparsity advantage; we discuss alternatives in Section 6.

## 6.2 Future Directions 

Learnable temporal fusion (e.g., sparse temporal attention or recurrent aggregation) could capture motion trajectories and improve detection of fast-moving drones. Multi-scale detection heads at strides 4, 8, and 16 with bidirectional feature pyramids could improve detection across varying target distances, and our preliminary multi-stride experiments (eliminating the FPN in favor of direct multi-stride prediction heads) suggest this is a viable direction for preserving sparsity while improving box regression, as it avoids the position-expanding transpose convolutions identified in Section 5.3. All results are reported on the FRED benchmark, the largest publicly available event-camera dataset for drone detection (629,832 annotated frames, 231 sequences, Prophesee EVK4 at 1280 × 720), recorded under diverse real-world con-ditions from daylight to complete darkness. While the sparse processing principles are sensor-agnostic (the architecture requires only coordinate–feature pairs from any voxelization), cross-class and cross-dataset benchmarking on additional event-camera datasets is an important direction for future work, as it would validate the generality of the fully sparse paradigm beyond single-class drone detection. 

## 6.3 Ethical Considerations 

Drone detection has dual-use implications: it can protect critical infrastructure but could also enable surveillance. In infrastructure protection contexts, false negatives (undetected drones) carry disproportionate risk; our error forensics (Section 5.2) reveals that 71% of current failures are spatially proximate to true targets (IoU > 0), suggesting that recall-oriented post-processing could address the most safety-critical failure mode. We advocate for transparent evaluation protocols and deployment guardrails that account for the asymmetric consequences of false positives and false negatives. 

## 6.4 Reproducibility 

All architecture code, training configurations, evaluation scripts, and the error forensics pipeline are publicly available at https://github.com/INQUIRELAB/ SparseVoxelDet. The FRED dataset is publicly available. Training on a single GPU and reporting multi-seed results makes the work accessible without large compute infrastructure and statistically grounded. 

# 7 Conclusion 

We presented SparseVoxelDet, a fully sparse object detector for event cameras that performs all computation, from backbone feature extraction through feature pyramid 24 fusion to the detection head, exclusively on occupied voxel positions using 3D sparse convolutions. On the FRED drone detection benchmark, SparseVoxelDet achieves 83.38% mAP@50 while processing only ∼14,900 active voxels per frame at 640 × 640, roughly 28 × fewer than the 409,600 pixels processed by the dense YOLOv11 baseline (87.68% mAP@50). The sparse representation achieves 858 × GPU memory compres-sion and 3,670 × storage reduction relative to the equivalent dense 3D voxel tensor, with voxel counts that remain nearly constant across sensor resolutions, eliminating the O(H × W ) scaling of dense methods. Our systematic error forensics reveals that the 4.3-point mAP@50 gap to the YOLOv11 baseline is dominated by localization near-misses (71% of failures), not missed detections. At a slightly relaxed IoU threshold of 0.40, SparseVoxelDet recovers to 89.26% mAP, confirming that the model detects drones in nearly every frame but produces bounding boxes that are slightly too loose for the strict IoU ≥ 0.50 criterion. The wider gap at mAP@50:95 (39.23% vs. 49.25%) further confirms that box regres-sion precision is the primary bottleneck. The model achieves 91.9% recall at IoU ≥ 0.50 (rising to 95.4% at IoU ≥ 0.40) with well-calibrated confidence scores. Multi-seed evaluation confirms training stability across random initializations ( ±0.16 pp at 640 2,

±0.14 pp at native resolution). These findings suggest that the sparse structure of event-camera data, exploited extensively by the LiDAR community but still underused in event-camera detection, is not only conceptually elegant but practically important: representation cost can scale with scene dynamics rather than sensor resolution. As event cameras improve in resolution and move toward edge deployment, fully sparse architectures like Sparse-VoxelDet provide a concrete path toward neuromorphic-native detection systems that preserve temporal detail without paying dense-resolution cost at every stage. Our mea-surements show that active voxel counts increase by only 9% when sensor resolution grows 2 .25 ×, a property unattainable by dense pipelines. The identification of box regression precision as the specific bottleneck, coupled with the strong detection recall (91.9%), provides a clear and focused target for future architectural improvement. More broadly, event cameras and sparse 3D architectures are natural complements: the sensor emits sparse, asynchronous signals, and the model computes only where those signals exist. This alignment opens a high-upside direction for the field, from high-resolution aerial monitoring to always-on, power-constrained edge intelligence, where dense pipelines are often bottlenecked by memory movement rather than recog-nition capacity. By establishing a fully sparse baseline and isolating its dominant failure mode, this work sets up the next phase of progress as an optimization problem with a clear target: retain sparse efficiency while tightening localization precision.
